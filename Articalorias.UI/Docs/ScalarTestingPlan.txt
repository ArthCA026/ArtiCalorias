SCALAR API TESTING PLAN — ARTICALORIAS BACKEND
================================================================================

Your API documentation UI is Scalar (not classic Swagger UI, but it works the same
way). When you launch the app it opens at /scalar/v1. All endpoints have a "Try it"
button and the JWT Authorize padlock.

================================================================================
PREREQUISITE — Set your JWT secret
================================================================================

Your appsettings.json has a placeholder key. It is 48 characters so it WILL work
for local testing as-is. But if you prefer a real secret, run this once in a terminal:

    dotnet user-secrets set "Jwt:SecretKey" "MySuperSecret-Articalorias-Key-MinLength32!!"

OpenAI note: The parse-food endpoint calls the OpenAI API. If you want to test it,
also run:

    dotnet user-secrets set "OpenAI:ApiKey" "sk-your-real-key-here"

If you skip this, everything else works — only the AI parsing endpoint will fail.

================================================================================
STEP 0 — Launch the app
================================================================================

Press F5 in Visual Studio (or dotnet run). The browser opens at:

    https://localhost:7205/scalar/v1

You will see all 6 controllers listed on the left sidebar. Each endpoint has a
"Try it" button that lets you send requests.

================================================================================
STEP 1 — Health Check (no auth required)
================================================================================

These two endpoints have no [Authorize], so test them first to verify your database
connection.

---- 1a. GET /api/health ----

  Click Try it -> Send
  Expected: 200 OK with { "status": "Healthy", "timestamp": "..." }
  Verifies: The API is running.

---- 1b. GET /api/health/db ----

  Click Try it -> Send
  Expected: 200 OK with { "status": "Connected", "tableRowCounts": { "users": 0, ... } }
  Verifies: SQL Server connection works and all 10 tables exist.
  If it fails: Check your connection string in appsettings.json and make sure
  SQL Server Express is running.

================================================================================
STEP 2 — Register a user
================================================================================

---- POST /api/auth/register ----

Click Try it, paste this body and Send:

    {
      "username": "arturo",
      "email": "arturo@test.com",
      "password": "TestPass123!"
    }

  Expected: 200 OK with { "token": "eyJhbG...", "username": "arturo", "userId": 1 }
  Verifies: User creation, password hashing, JWT generation.
  COPY THE TOKEN VALUE — you will need it in the next step.

  Try sending it again with the same username — you should get an error (duplicate
  username).

================================================================================
STEP 3 — Login
================================================================================

---- POST /api/auth/login ----

    {
      "username": "arturo",
      "password": "TestPass123!"
    }

  Expected: 200 OK with a new JWT token.
  Copy this token — this is the one you will use going forward.

  Test with a wrong password to verify you get an error.

================================================================================
STEP 4 — Authorize in Scalar
================================================================================

This is the key step — all remaining endpoints require JWT.

  1. At the top of Scalar, look for the Authentication section (or the lock icon).
  2. Select Bearer as the type.
  3. Paste the token you copied from Step 3 (just the token string, WITHOUT the
     word "Bearer").
  4. Click Apply or Save.

From now on, every request will include the Authorization: Bearer <token> header
automatically.

How to verify: Try any [Authorize] endpoint. If you get 401 Unauthorized, the
token was not applied. If you get a real response, you are authenticated.

STEP 5 — Create your user profile (onboarding)
================================================================================

---- 5a. GET /api/userprofile ----

  Click Try it -> Send
  Expected: 404 Not Found with "Profile not found. Complete onboarding."
  Verifies: New users don't have a profile yet.

---- 5b. PUT /api/userprofile ----

This is the "create or update" endpoint. Paste this body:

    {
      "currentWeightKg": 80.5,
      "heightCm": 175,
      "bmrKcal": 1800,
      "bodyFatPercent": 18.5,
      "dailyBaseGoalKcal": -500,
      "proteinGoalGrams": 150,
      "autoCalculateProteinGoal": false
    }

  Expected: 200 OK with the full UserProfileResponse including
  isOnboardingCompleted: true.
  Verifies: Profile creation, default values, onboarding flag.

---- 5c. GET /api/userprofile (again) ----

  Expected: 200 OK with all the profile data you just saved.
  Verifies: Read-back works.

---- 5d. Update the profile (optional) ----

Call PUT /api/userprofile again with a different weight to verify updates:

    {
      "currentWeightKg": 79.0,
      "heightCm": 175,
      "bmrKcal": 1780,
      "bodyFatPercent": 17.5,
      "dailyBaseGoalKcal": -500,
      "proteinGoalGrams": 150,
      "autoCalculateProteinGoal": false
    }

STEP 6 — Daily Log basics
================================================================================

---- 6a. GET /api/dailylog/2025-06-24 ----

================================================================================
STEP 8 — Recalculate the daily log
================================================================================

---- POST /api/dailylog/2025-06-24/recalculate ----

  No body needed.
  Expected: 200 OK with the full DailyLogResponse.
  Verify these fields are now populated:
    - totalFoodCaloriesKcal       — should match the remaining food (~412)
    - totalProteinGrams           — ~77
    - tefKcal                     — per-macro TEF (protein×4×0.25 + fat×9×0.02 + carbs×4×0.08 + alcohol×7×0.15)
    - totalDailyExpenditureKcal   — BMR + TEF + activity
    - netBalanceKcal              — food calories minus expenditure
    - dailyGoalDeltaKcal          — net balance compared to your -500 goal
    - caloriesRemainingToDailyTargetKcal — how much you can still eat
    - Weekly projection fields (weeklyTargetKcal, weeklyActualToDateKcal, etc.)

  This is the MOST IMPORTANT VERIFICATION — it tests the full 10-step
  recalculation pipeline.

SUMMARY — Testing Order Cheat Sheet
================================================================================

  #   Endpoint                              Auth?   What it proves
  --  ------------------------------------  ------  ----------------------------------
  1   GET /api/health                       NO      API is alive
  2   GET /api/health/db                    NO      DB connected, tables exist
  3   POST /api/auth/register               NO      User creation + JWT
  4   POST /api/auth/login                  NO      Login + token issuance
  5   (Authorize in Scalar)                 --      Paste token
  6   GET then PUT /api/userprofile         YES     Profile CRUD + onboarding
  7   GET /api/dailylog/{date}              YES     Auto-create daily log
  8   POST/GET/PUT/DELETE .../foods         YES     Food entry CRUD
  9   POST .../recalculate                  YES     10-step pipeline
  10  POST .../parse-food                   YES     OpenAI integration
  11  POST .../foods/batch                  YES     Batch confirm AI foods
  12  Template CRUD                         YES     Activity template catalog
  13  Activity entry CRUD                   YES     Logging exercise
  14  Recalculate again                     YES     Full pipeline with activities
  15  Dashboard                             YES     Composite response
  16  History endpoints                     YES     Range queries + summaries
  17  Bad payloads                          YES     Validation rules

Follow this order exactly and by Step 15 you will have verified every layer of
your backend. Once everything looks good, we move to the React frontend!
